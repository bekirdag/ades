# ades v0.1.0 Decisions

## 1. What exactly is `v0.1.0`?

Answer: `v0.1.0` is the first usable local product, not the final full platform. It includes a Python package, CLI, local HTTP service, pack manager, English text and HTML tagging, pack-aware entity extraction, lightweight linking, and stable JSON output.

## 2. Is `v0.1.0` local-first or server-first?

Answer: local-first. The product must work on a single machine after packs are pulled. Distributed workers, hosted infrastructure, and cluster-scale components are out of scope for this release.

## 3. What commands must exist in `v0.1.0`?

Answer: `ades pull <pack>`, `ades serve`, `ades tag <text>`, `ades status`, and `ades packs list`.

## 4. What is the canonical version target?

Answer: the first release target is `v0.1.0`. The local service API is versioned under `/v0` to match the pre-1.0 maturity level.

## 5. Are packs standalone or layered?

Answer: layered. Domain packs may depend on a smaller shared core language pack. `ades pull medical-en` should automatically install any required base pack dependency instead of duplicating the same base artifacts across every domain pack.

## 6. What manifest format should packs use?

Answer: JSON. Each pack ships with a machine-first `manifest.json` using manifest schema version `1`. JSON is the best fit for Python, npm tooling, checksum automation, and static registry hosting.

## 7. What should a pack manifest contain?

Answer: at minimum:

- `schema_version`
- `pack_id`
- `version`
- `language`
- `domain`
- `tier`
- `dependencies`
- `artifacts`
- `models`
- `rules`
- `labels`
- `min_ades_version`
- `sha256`

## 8. What archive format should pack artifacts use?

Answer: `.tar.zst`. It gives materially better compression for large local data than zip while staying practical to handle from Python.

## 9. Where must downloaded data live?

Answer: under `/mnt/githubActions/ades_big_data` only.

Recommended layout:

- `/mnt/githubActions/ades_big_data/packs/`
- `/mnt/githubActions/ades_big_data/models/`
- `/mnt/githubActions/ades_big_data/cache/`
- `/mnt/githubActions/ades_big_data/tmp/`
- `/mnt/githubActions/ades_big_data/registry/ades.db`
- `/mnt/githubActions/ades_big_data/logs/`

## 10. What local persistence should `v0.1.0` use?

Answer: `SQLite` only. Use one runtime registry database for installed-pack metadata and status, and allow each pack to ship its own SQLite lookup data for aliases, rules, or lightweight KB tables.

## 11. What local search/index mechanism should `v0.1.0` use?

Answer: deterministic `SQLite` tables plus exact alias and rule lookup. `SQLite FTS5` remains the first candidate upgrade if broader local candidate search becomes necessary, but it is not a shipped default requirement for the current `v0.1.0` baseline.

## 12. What service endpoints should exist in `v0.1.0`?

Answer:

- `GET /healthz`
- `GET /v0/status`
- `GET /v0/packs`
- `GET /v0/packs/{pack}`
- `POST /v0/tag`

Pack installation remains CLI-owned in `v0.1.0`.

## 13. What input types must `v0.1.0` support?

Answer: plain text and HTML string input. PDF and OCR are deferred. They add large dependencies and slow the first release without being essential to the pack-manager and local-service foundations.

## 14. What should the request and response contract look like?

Answer: keep it simple and stable.

Request:

- `text`
- `content_type` with `text/plain` or `text/html`
- `pack`
- optional `options`

Response:

- `version`
- `pack`
- `language`
- `entities`
- `topics`
- `warnings`
- `timing_ms`

Entity matches may also carry `confidence`, `relevance`, deterministic provenance, and lightweight link metadata. Topic matches may carry a score, evidence count, and the contributing entity labels.

## 15. Which NLP stack should `v0.1.0` use by default?

Answer:

- the current default runtime stays deterministic-only
- extraction uses pack-provided rules, aliases, labels, and exact lookup over local `SQLite` metadata
- `spaCy`, `GLiNER`, and `FlashText` remain candidate upgrades after `v0.1.0`, not default runtime dependencies for the current release

This keeps the first release small, local-first, and easy to install on normal developer machines.

## 16. How should models be stored and loaded?

Answer: store models once under `/mnt/githubActions/ades_big_data/models/`, let packs reference them, and load them lazily on first use. Do not duplicate the same large model inside multiple packs.

## 17. What should entity linking look like in `v0.1.0`?

Answer: lightweight alias and dictionary-based linking with confidence tiers. Full REL or OpenTapioca integration is deferred until the local pack/runtime model is proven useful.

## 18. What should topic classification look like in `v0.1.0`?

Answer: lightweight pack-driven tagging only. Use deterministic entity evidence to derive topic scores, evidence counts, and contributing label summaries. Heavy zero-shot or few-shot topic classifiers are deferred unless they become necessary after the local `v0.1.0` baseline is proven useful.

## 19. Is relation extraction in scope for `v0.1.0`?

Answer: no. It is deferred. The first release should ship strong entity extraction and pack-aware enrichment before taking on relation extraction complexity and licensing risk.

## 20. Which dependencies are allowed in the default `v0.1.0` runtime?

Answer: commercially safe defaults only. Prefer MIT, BSD, Apache 2.0, or similarly safe licenses for the default shipped path. Do not make REBEL or another non-commercial dependency part of the initial runtime.

## 21. What packaging strategy should be used?

Answer:

- canonical Python runtime package: `ades`
- npm distribution: `ades-cli`
- installed command in both cases: `ades`

The npm package is a thin wrapper around the Python runtime bootstrap, not a reimplementation of the enrichment engine.

## 22. How should packs be published?

Answer: through a static registry. `v0.1.0` should use a versioned registry JSON plus HTTPS-hosted artifacts with checksums. There is no need for a custom central pack-management API in the first release.

## 23. What should the default local service binding be?

Answer: `127.0.0.1:8734` by default, configurable through CLI flags or environment variables such as `ADES_HOST` and `ADES_PORT`.

## 24. What should be considered out of scope for `v0.1.0`?

Answer:

- distributed workers
- Redis and Celery
- Postgres and OpenSearch
- vector databases
- PDF OCR
- multilingual runtime support
- full relation extraction
- heavyweight entity-linking stacks

## 25. How do local and production modes differ?

Answer:

- `v0.1.0` ships the `local tool`
- the local tool uses `SQLite`
- the later `production server tool` will use `PostgreSQL`
- the repo should keep explicit placeholders and seams for that future server path, but the PostgreSQL implementation is out of scope for `v0.1.0`

## 26. What test coverage is mandatory on every development step?

Answer: every development step must include unit, component, integration, and API tests. The exact scope can stay small while the product is early, but all four layers must be represented for each new feature or structural change.
